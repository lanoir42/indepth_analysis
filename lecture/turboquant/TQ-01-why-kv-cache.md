# TQ-01: KV Cache — 왜 이것이 병목인가?

> **목표**: Transformer의 Self-Attention에서 KV Cache가 왜 필요한지, 왜 메모리를 폭발시키는지 이해
> **선수 지식**: 행렬 곱셈, 벡터 내적
> **소요 시간**: 40분

---

## 1. Transformer가 글을 읽는 방법

### 핵심 질문

> "ChatGPT가 당신의 긴 질문을 읽을 때, 내부적으로 무슨 일이 일어나는가?"

Transformer는 **Self-Attention** 메커니즘으로 작동합니다. 모든 토큰이 다른 모든 토큰을 "참조"합니다.

### Attention 공식

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) \cdot V$$

세 가지 행렬이 등장합니다:

| 행렬 | 이름 | 역할 | 비유 |
|------|------|------|------|
| $Q$ | **Query** | "내가 찾고 싶은 것" | 도서관에서의 검색어 |
| $K$ | **Key** | "각 토큰이 가진 인덱스" | 책의 색인 |
| $V$ | **Value** | "실제 내용" | 책의 본문 |

**직관**: 새 토큰을 생성할 때, Query로 모든 Key를 검색하고 → 가장 관련 높은 Value를 가져옵니다.

### softmax가 하는 일

$$\text{softmax}(z_i) = \frac{e^{z_i}}{\sum_j e^{z_j}}$$

내적 $QK^T$의 결과를 **확률**로 변환합니다. 큰 내적 → 높은 가중치 → 그 토큰에 더 많은 주목(attention).

---

## 2. 왜 K와 V를 저장해야 하는가?

### 자기회귀 생성 (Autoregressive Generation)

ChatGPT는 한 번에 한 토큰씩 생성합니다:

```
입력:  "서울의 날씨는"
출력:   ↓ ↓ ↓ ↓ ↓
       "오" → "늘" → "맑" → "습" → "니다"
```

**"맑"을 생성할 때**, 모델은 "서울의", "날씨는", "오", "늘"의 K, V를 모두 참조해야 합니다.

### 매번 재계산 vs 캐싱

**방법 1: 매번 재계산** — "맑"을 생성할 때 앞의 4개 토큰 전부 다시 계산
- 비용: $O(n^2)$ — 토큰이 늘어날수록 **제곱으로** 증가
- 128K 토큰이면 $128{,}000^2 = 16.4 \times 10^9$ 연산을 매 토큰마다 수행

**방법 2: KV Cache** — 이전 토큰의 K, V를 **저장**해두고 재사용
- 새 토큰의 Q만 계산하면 됨
- 비용: $O(n)$ — 선형

→ **KV Cache는 추론 속도를 극적으로 높이는 핵심 최적화입니다.**

## 3. KV Cache가 먹는 메모리

### 메모리 계산 공식

$$\text{KV Cache} = 2 \times L \times H \times d \times n \times b$$

각 변수의 의미:

| 변수 | 의미 | 예시 (Llama-3 70B) |
|------|------|-------------------|
| $2$ | K와 V 각각 | 2 |
| $L$ | 레이어 수 | 80 |
| $H$ | 어텐션 헤드 수 | 64 |
| $d$ | 헤드 차원 | 128 |
| $n$ | 시퀀스 길이 (토큰 수) | 128,000 |
| $b$ | 정밀도 (바이트) | 2 (FP16) |

### 실제 계산

$$2 \times 80 \times 64 \times 128 \times 128{,}000 \times 2 \approx 335 \text{ GB}$$

비교: 모델 가중치 자체는 ~140 GB (FP16 기준).

> **KV Cache가 모델 자체보다 2.4배 더 크다!**

### 그림으로 보기

```
                 짧은 대화 (1K 토큰)        긴 문서 분석 (128K 토큰)
              ┌──────────────────┐      ┌──────────────────┐
              │                  │      │                  │
   가중치      │  ████████████    │      │  ██              │
   (140 GB)   │  ████████████    │      │                  │
              │                  │      │                  │
   KV Cache   │  ██              │      │  ████████████    │
   (? GB)     │                  │      │  ████████████    │
              │                  │      │  ████████████    │
              │                  │      │  ████████████    │
              └──────────────────┘      └──────────────────┘
                  KV ≈ 2.6 GB              KV ≈ 335 GB
```

## 4. 왜 가중치 양자화만으로는 부족한가?

### 기존 연구의 초점

GPTQ, AWQ, SqueezeLLM 같은 기존 양자화 기법은 **모델 가중치**를 4-bit로 줄입니다:

$$\text{가중치 (FP16)} = 140 \text{ GB} \xrightarrow{\text{4-bit}} 35 \text{ GB}$$

하지만 KV Cache는 그대로 335 GB!

### 긴 컨텍스트 시대의 문제

2024년 이후 LLM은 **100K~10M 토큰** 컨텍스트를 지원합니다. 이 길이에서는:

$$\frac{\text{KV Cache}}{\text{총 메모리}} \to 1 \quad (n \to \infty)$$

컨텍스트가 길어질수록 **KV Cache가 메모리의 거의 전부**를 차지합니다.

### TurboQuant의 목표

> **KV Cache를 3~3.5 bit로 양자화하여 335 GB → 49~56 GB로 줄이되, 정확도를 전혀 잃지 않는다.**

이것이 다음 강의부터 다룰 핵심 과제입니다.

## 5. 실감하기: NVIDIA H100 한 장으로 무엇이 가능한가?

NVIDIA H100 GPU의 메모리: **80 GB**

| 설정 | 가중치 | KV Cache | 가능? |
|------|--------|----------|------|
| Llama-3 70B, FP16, 128K | 140 GB | 335 GB | ❌ (475 GB 필요) |
| Llama-3 70B, 4-bit 가중치, FP16 KV | 35 GB | 335 GB | ❌ (370 GB) |
| Llama-3 70B, 4-bit 가중치, **3.5-bit KV (TurboQuant)** | 35 GB | **49 GB** | ✅ **(84 GB — 1장에 가능!)** |

> **TurboQuant 하나로, 수십 장의 GPU가 필요했던 작업을 1장으로 줄일 수 있습니다.**

## 6. 이 강의에서 기억할 것

1. **KV Cache**는 Transformer가 이전 토큰의 Key와 Value를 저장하는 메모리
2. 컨텍스트가 길어지면 KV Cache가 **모델 가중치보다 훨씬 커진다**
3. 기존 양자화 기법은 **가중치**를 줄이지만 KV Cache는 건드리지 않음
4. TurboQuant는 **KV Cache 자체를 3-bit로 줄이는 것**이 목표
